# 狙击外星人升级版

这是一个用 AIGC 推进游戏生产流程的实验项目。项目以“狙击外星人”式的识别射击玩法为起点，通过 MCP 调用 Godot 迭代原型，用混元 3D 和其他 AIGC 工具补齐资产，把镜头、UI、状态流转、配置规范和文档规范落成了一个可运行的游戏原型。

这份仓库不只是一个 Demo，更是一份关于“游戏如何被生产出来”的工程记录。它把玩法、美术、系统、状态、配置和协作规范放在同一个项目里，用一个可运行项目把多管线协作的关键问题串了起来。

## 为什么做这个项目

这个项目关注的是识别型射击玩法里的“判断快感”：乐趣不只是射击，而是识别、怀疑、确认和误判。玩家不是看见目标就开枪，而是要在人群里找出“不太对劲”的角色，再用镜头、缩放、屏息和反馈完成判断。

这个题材也很适合做 AIGC 驱动实验，因为它要求玩法框架、角色、掩体、武器、镜头、UI 和反馈一起工作。项目想验证的不是“AI 能不能出图”，而是“AI 能不能进入游戏生产流程，参与原型、资产、文档和工程规范的协作”。

## 玩法从哪里来

这个项目的玩法参考来自“狙击外星人”式的识别射击框架，但我没有把重点放在传统 FPS 的反应速度上，而是放在“观察 - 判断 - 确认”的识别快感上。玩家在 3D 城市场景里搜索目标，通过 2.5D 镜头缩短信息处理路径，再用局部特征判断谁是伪装外星人，谁只是制造假线索的平民。

所以这个项目最核心的闭环不是“开枪”，而是：

`扫视场景 -> 锁定可疑角色 -> 放大瞄准 -> 屏息稳定 -> 射击判定 -> 命中 / 误伤 / 挡弹反馈 -> 继续搜索或进入结算`

这条闭环后来变成了整个工程实现的轴心。镜头怎么推、角色怎么设计、反馈怎么给、状态怎么流转、关卡怎么配，最后都要回到这条主循环上。

## 设计框架与落地方式

### 用 MCP 驱动 Godot 原型生成

这套方案没有采用“先把文档写满，再开始开发”的路径，而是通过 MCP 去调度 Godot，让玩法原型和工程结构一起长出来。它最重要的价值不是“省时间”，而是把原本停留在想法层的内容快速压到可验证状态。

当玩法、界面、场景和状态已经能在 Godot 里跑起来，镜头是否顺手、状态切换是否自然、反馈是否过量、关卡节奏是否成立都会更早暴露出来。这种方式的核心价值在于：原型不是演示品，而是设计判断的试验台。

### 用混元 3D 和 AIGC 补齐美术资产

在美术资产上，AIGC 没有被当成“最后出图”的工具，而是被放进资产生产管线。混元 3D、参考图生成、几何基准、贴图校准、角色变体、武器预览和占位素材，拼起来才构成一个能进入工程的资产流程。

这条管线强调的不是“先把图做出来”，而是“先让资产具备接入能力”。资产不是“有了就行”，而是必须可替换、可挂接、可命名、可验证。只要进入引擎，它就不再是一个孤立模型，而是一个要和挂点、状态、反馈、材质和镜头一起工作的生产对象。

### 把 UI、镜头与交互落地

页面设计和镜头设计在这里被当成玩法信息结构的一部分来处理。主菜单、战斗 HUD、结算面板、武器库、商店、教程页，看起来是 UI，但真正影响体验的是它们和战斗状态、玩家意图之间的关系。

镜头同样如此。这里不是单纯做一个“能缩放的相机”，而是在处理搜索态、瞄准态、屏息态、命中反馈和结算过渡时，让镜头成为玩法理解的一部分。换句话说，镜头不是摄影机功能，而是玩法信息结构的一部分。

### 把配置、文档和资产管理工程化

如果一个项目只停留在“原型能跑”，它仍然只是一个实验。要进入持续迭代阶段，就必须把关卡、资产、字段、命名、挂点、验证规则和文档口径做成工程化资产。

这也是仓库里为什么会出现大量规范文档、映射说明、验证结果和资产合同表。它们不是附属物，而是支撑持续迭代的骨架。一旦项目涉及玩法、美术、程序和工具链协作，这些规范就不再是“锦上添花”，而是防止系统失控的前提。

## 设计原则与实践结论

### 玩法设计要围绕判断链路

识别型射击玩法真正成立，依赖的不是单点机制，而是识别成本、误判代价、反馈语义和玩家信息获取路径。一个玩法成立，不是因为概念好听，而是因为玩家能在每一次判断里感受到明确的因果关系。

### 状态设计决定系统是否可信

射击、瞄准、缩放、屏息、命中、误伤、挡弹、结算、返回，这些内容看上去是很多小功能，但一旦把它们串起来，本质上就是在设计状态机。状态设计不是为了代码优雅，而是为了让玩家感知到世界是连续的、反馈是可信的、系统是有秩序的。

### 配置化是持续迭代的前提

关卡设计一旦想持续迭代，就不能靠手改逻辑顶着做。目标类型、行为模式、伪装强度、移动范围、暴露节奏、关卡目标数、反馈参数，如果不能通过配置驱动，后面的调优成本会急剧上升。

配置化的价值不是方便策划，而是让系统具备持续迭代能力。它决定了一个项目后面是越改越快，还是越改越乱。

### 工程规范是多管线协作的底座

文档规范、资产管理、命名规则看起来像辅助工作，但一旦没有这些东西，多管线协作就会立刻失真。AIGC 资产接进来以后，命名、挂点、材质、替换顺序、验证口径如果不统一，系统很快就会失控。

也正因为这样，这个项目最核心的价值，不只是一个玩法原型，而是一套关于游戏生产过程的组织方式：玩法怎么设计、状态怎么组织、资产怎么接入、配置怎么演进、文档怎么支撑协作。

## 这个项目为什么值得看

这个项目最值得看的，不是它做出了一个识别射击 Demo，而是它把前沿 AI 游戏生成里真正重要的问题落到了工程细节上：玩法主循环怎么成立，资产怎么接入引擎，状态怎么组织，关卡怎么配置化迭代，文档和规范怎么支撑多管线协作，以及 AI 分析如何进入交互闭环。

## 本地运行

1. 安装 `Godot 4.x`
2. 安装 `Git LFS`
3. 克隆仓库后执行：`git lfs install && git lfs pull`
4. 用 Godot 打开 `狙击外星人升级版/`
5. 等待资源导入完成后运行主工程

## 可选：启用 LLM 分析

战斗分析功能可以接本地 LLM 代理。这部分是可选实验，不配置也能正常游玩主流程。

1. 复制 `llm_proxy/.env.example` 为 `llm_proxy/.env`
2. 在本地 `.env` 中填写：
   - `ARK_API_KEY`：你自己的 API Key
   - `ARK_MODEL`：你要使用的 endpoint / 模型 ID
3. 在 `llm_proxy/` 目录运行 `python app.py`
4. 再启动 Godot 主工程

注意：

- `.env` 只能本地使用，不能上传到 GitHub
- 仓库里的配置文件不再保存真实 key
- 如果你不需要 LLM 分析，可以直接跳过这部分
