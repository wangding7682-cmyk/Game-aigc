## 文档范围

这份文档只回答 3 个问题：

1. 现在这个项目到底在做什么
2. 当前可运行版本的核心玩法和边界是什么
3. 哪些内容已经不是当前设计基线

- `Canvas/Phaser/H5` 作为当前实现方案

## 当前项目定义

### 核心一句话

玩家在 `2.5D 搜索式镜头` 下，于 `3D 城市场景` 中识别伪装外星人与平民差异，完成观察、放大、屏息、射击、复盘这一条闭环。

### 当前产品形态

| 维度 | 当前基线 |
|---|---|
| 引擎 | `Godot 4.x` |
| 视觉形态 | `3D 画面 + 2.5D 识别射击玩法` |
| 当前主模式 | `PVE 三关验证 + 教程 + 基础成长` |
| 平台状态 | 桌面可运行，微信/抖音保留平台适配骨架 |
| PVP 状态 | 保留大厅、房间与网络控制器骨架，当前不是主验收范围 |
| 资源形态 | `3D 占位资产 + 工程化规则 + 部分正式导入资产` |

### 当前设计目标

当前阶段不是把整个产品面铺满，而是先把下面 5 件事做实：

1. 搜索与识别是否成立
2. 放大与屏息后的手感是否稳定
3. 误判惩罚与复盘是否让玩家理解“为什么错”
4. 扫描、加时、挡弹、命中等反馈是否清楚
5. 文档规范、资源规则、工程实现是否能互相对上

---

## 当前玩法闭环

### 单局主循环

`扫视场景 -> 锁定可疑角色 -> 放大瞄准 -> 屏息稳定 -> 射击判定 -> 命中/误判/挡弹反馈 -> 继续搜索或进入结算`

### 当前核心动词

1. `观察`
2. `识别`
3. `瞄准`
4. `射击`
5. `复盘`

### 当前镜头与输入原则

当前工程已经按下面这套口径实现：

- `按住拖动` 扫视场景
- `滚轮中心缩放`，尽量保持缩放中心稳定
- 进入瞄准后保留当前观察状态，不做粗暴重置
- 屏息只改变稳定性，不直接替玩家锁敌
- 搜索态和瞄准态都要保留“玩家自己做判断”的空间

## 当前 PVE 设计范围

### 当前已纳入基线的内容

| 模块 | 当前口径 |
|---|---|
| 关卡 | `3` 关，走配置驱动 |
| 教程 | 有，且与首关衔接 |
| 目标类型 | `静止伪装 / 缓慢移动 / 弱点窗口` |
| 干扰对象 | `平民`，带受控假线索 |
| 道具 | `扫描雷达 / 时间延长` |
| 失败条件 | 时间耗尽或误伤达到上限 |
| 结算 | 有基础胜负结算和金币沉淀 |
| 复盘 | 有 `killcam` 与 `误判回放` 方向 |

### 当前未纳入基线的内容

以下内容即使在历史文档里出现过，也不再视为当前设计承诺：

- 大量 PVE 地图池
- 动态昼夜天气系统
- Boss 链路
- Battle Pass
- 段位赛季
- 复杂道具组合系统
- 完整充值和广告数值规划

这些内容后续如果重启，必须重新进入设计评审，不能直接沿用旧文档口径。

## 目标识别设计

### 身份划分

当前只保留两类核心身份：

1. `外星人目标`
2. `平民干扰项`

### 外星人识别原则

外星人不再依赖“大量夸张表演”去暴露身份，而是以 `最小特征方案` 为基线：

- 轮廓先给出“不太对劲”的感觉
- 中景靠材质与结构进一步确认
- 近景只补一个简单弱点开关态

### 当前稳定线索

外星人优先使用这些信号：

1. `头部外壳过紧`
2. `肩袖连接偏硬`
3. `手套与手臂像一体件`
4. `腰侧细亮线短暂显露`

### 平民假线索原则

平民可以误导，但不能变成“另一种外星人”。

允许存在的假线索：

1. `发卡或耳饰反光`
2. `胸针或拉链头反光`
3. `围巾或衣领晃动`
4. `步频或配饰造成的短时错觉`

不允许把平民做成持续危险红光、持续异常肩线或稳定弱点结构。

## 当前战斗规则

### 命中结果

当前战斗结果只区分这几类：

| 结果 | 含义 |
|---|---|
| `hit` | 正确命中目标 |
| `damage` | 命中目标但未击杀 |
| `wrong_hit` | 误伤平民 |
| `blocked` | 被掩体或障碍挡住 |
| `miss` | 未命中有效对象 |

### 当前反馈语义

| 反馈 | 语义 |
|---|---|
| 绿色 | 正确命中 / 成功 |
| 红色 | 误判 / 危险 |
| 青蓝 | 扫描 / 搜索辅助 |
| 冷蓝或中性亮色 | 挡弹 / 掩体反馈 |
| 白蓝 | 慢动作 / 回放强调 |

### 当前道具口径

当前只保留两个基础道具：

1. `扫描雷达`
   - 作用：临时强化目标识别
   - 当前表现：世界空间扫描波、目标高亮、提示优先目标
2. `时间延长`
   - 作用：补足容错
   - 当前表现：直接增加剩余时间并同步 HUD

## 当前关卡与成长

### 关卡配置原则

当前关卡优先由配置决定，而不是把所有参数写死在逻辑里。

每个刷点条目当前至少可控制：

- `actor_kind`
- `behavior_type`
- `position`
- `tutorial_primary`
- `disguise_strength`
- `move_range`
- `move_speed`
- `reveal_cycle_sec`
- `reveal_window_sec`
- 搜索提示与假线索相关字段

### 玩家手感与关卡难度分层

当前项目明确把两件事拆开：

1. `玩家手感层`
   - 解决“操作顺不顺手”
2. `关卡配置层`
   - 解决“这一关难不难、目标怎么出现”

这意味着：

- 玩家设置不应该直接改坏关卡口径
- 策划调难度不应该去改玩家基础操作体验

### 当前成长范围

当前只保留最基础的成长闭环：

- 关卡结算发金币
- 金币进入库存
- 下一局能带着升级后的手感继续验证

不再把大规模数值池、赛季成长、长线商业化循环写进当前设计基线。

## 当前工程映射

### 当前设计必须能映射到工程

当前设计不是抽象愿景，而是要能落回现有工程节点与脚本上。

核心脚本映射如下：

| 能力 | 现有工程入口 |
|---|---|
| 3D 相机与缩放 | `camera_controller_3d.gd` |
| 3D 战斗规则 | `battle_core_3d.gd` |
| 3D 目标行为 | `pve_target_controller_3d.gd` |
| 3D 战斗反馈 | `visual_feedback_3d.gd` |
| 3D 武器表现 | `weapon_renderer_3d.gd` |
| 测试中心 | `ui_panel_test_center.gd` |
| 教程流程 | `tutorial_flow_controller.gd` |
| PVP 骨架 | `pvp_lobby_controller.gd`、`pvp_network_room_controller.gd` |

### 当前场景结构原则

战斗场景继续沿用并强化下面这套结构：

```text
BattleRoot3D
├─ LevelRoot
├─ ActorRoot
├─ DecalRoot
├─ FxRoot
├─ AudioRoot
└─ UiRoot
   ├─ HudLayer
   ├─ PopupLayer
   ├─ GuideLayer
   ├─ LoadingLayer
   └─ DebugLayer
```

这套结构是当前工程、美术挂载和调试入口之间的共用基线。

## 当前美术与资产基线

### 美术方向

当前视觉基线保持为：

- `都市夜战`
- `轻科幻`
- `软 3D 卡通体块`
- `可读性优先`

当氛围感和玩法可读性冲突时，统一优先玩法可读。

### 当前资产策略

当前项目不是“还没接 3D”，而是已经进入：

`3D 占位资产可运行 + 规则文档成型 + 部分资产已导入主工程`

当前可确认已进入工程链路的类别包括：

1. 角色占位
2. 武器占位
3. 掩体与街区基础块
4. 枪口火焰与反馈序列帧占位
5. 扫描、命中、误判、挡弹的世界空间反馈

### 当前仍然存在的资源工作

还未完成的不是“有没有占位”，而是下面这些正式化工作：

1. 正式量产模型替换
2. 正式材质分级与 LOD
3. 贴花锚点与真实资产挂点统一
4. 动作体系与骨骼规则进一步收口
5. PVE 与预留 PVP 资产规范继续对齐

## PVP 当前定位

### 当前口径

PVP 在当前工程里已经有骨架，但它现在的定位是：

`保留接口与控制器结构，作为后续扩展方向，不作为当前主设计验收范围。`

### 当前允许保留的内容

- 大厅控制器
- 房间控制器
- 网络管理器
- 本地烟雾或 mock 验证入口

### 当前不再写入主基线的内容

- 完整排位体系
- 完整赛季制
- 真联网对战的数值、地图池和商业化结构

## 业务化验证口径

### 玩家体验验证

当前版本至少要能稳定回答下面这些问题：

1. 新玩家能否在首关快速理解“看哪里、怎么开枪”
2. 玩家能否区分“真线索”和“假线索”
3. 玩家在误伤之后，能否从回放里理解自己错在哪
4. 扫描是否真的降低了找目标时间
5. 掩体挡弹时，玩家是否一眼看懂“不是没打中，而是被挡住了”

### 工程验证

当前工程已明确支持一组贴近业务链路的烟雾验证：

1. 主流程烟雾
2. 下一关烟雾
3. 完整集成烟雾
4. 路由守卫烟雾
5. 3D 占位烟雾

这些验证的意义不是为了凑测试数量，而是确认：

- 关卡能进
- 玩法能跑
- 结果能结算
- 资源能显示
- 路由不会在关键节点断掉

## 已移除的旧信息

本次校准后，以下信息明确从当前设计基线移除：

- 当前技术栈是 `Canvas/Phaser`
- 当前产品已经进入完整 `IAA + IAP` 运营规划阶段
- 当前主验收目标包含 `Boss / BP / 赛季 / 段位`
- 当前还是“2D 代理验证，3D 尚未接入”
- 当前必须先做完整真联网 PVP 才能继续推进

如果后续要恢复这些方向，必须以新版本设计文档重新进入，而不是回退引用旧版内容。

---

## 一句话结论

当前《狙击大师》的真实设计基线是：`围绕 Godot 3D MVP 的 PVE 识别射击闭环，把识别、手感、反馈、资源规则和工程实现先对齐做实，其他大系统暂时全部降级为预留方向。`
