# 关卡复盘看板说明

## 文档用途

这份说明只解决两件事：

1. 统一 `PVE 1~3 关` 的埋点字段口径
2. 让关卡复盘时能直接判断问题在 `识别`、`操作`、还是 `决策`

它不是泛泛的埋点清单，而是给当前 MVP 关卡调优直接使用的看板说明。

## 日志来源

当前埋点由 `AnalyticsService` 写入本地日志，格式是 `jsonl`，一行一个事件。

当前复盘主要看两类来源：

- 过程日志：`user://analytics_events.jsonl`
- 烟雾测试结果：`user://integration_smoke_result.txt`

说明：

- `user://` 是 Godot 的项目用户目录，不同机器上实际落盘路径可能不同
- 复盘时优先看事件字段，不要只看控制台打印

## 看板结构

建议把关卡复盘看板固定成 5 张卡，避免每次复盘都重新想怎么解释数据。

### 卡 1 关卡进入

用途：确认玩家真的进到了哪一关，而不是只看菜单点击量。

核心事件：

- `level_entered`

关键字段：

| 字段 | 含义 |
|------|------|
| `level_id` | 当前关卡编号 |
| `level_name` | 当前关卡名称 |
| `tutorial_active` | 是否处于教程锁步阶段 |
| `target_goal` | 目标总数 |
| `moving_targets` | 移动目标数 |
| `weakpoint_targets` | 弱点目标数 |

复盘问题：

- 第 1 关进入量是否明显高于后两关
- 第 2、3 关是否存在进入量正常但完成量明显掉队

### 卡 2 教程与首枪

用途：判断第 1 关问题到底是“没看懂”，还是“看懂了但第一枪成本高”。

核心事件：

- `tutorial_started`
- `tutorial_completed`
- `first_hit_recorded`

关键字段：

| 字段 | 含义 |
|------|------|
| `step_index` | 教程起始步骤编号 |
| `step_title` | 当前步骤标题 |
| `elapsed_time` | 教程完成耗时 |
| `first_hit_elapsed_sec` | 本局首次正确命中耗时 |

核心指标：

- 教程完成率
- 首次命中耗时

判断口径：

- 教程完成率低：优先检查锁步顺序、提示文案、操作门槛
- 教程完成率高但首枪耗时长：优先检查可疑特征是否太弱、镜头观察成本是否太高

### 卡 3 射击质量

用途：把“这关不好打”拆成更具体的命中、打空、误伤结构。

核心事件：

- `shot_fired`
- `battle_finished`

关键字段：

| 字段 | 含义 |
|------|------|
| `result` | 本次射击结果，常见值有 `hit`、`miss`、`wrong_hit`、`blocked` |
| `shot_count` | 当前累计射击数 |
| `hit_count` | 当前累计正确命中数 |
| `wrong_hit_count` | 当前累计误伤数 |
| `accuracy` | 本局命中率 |

核心指标：

- `miss_rate`
- `wrong_hit_rate`
- `accuracy`

判断口径：

- `miss` 高：更像跟枪压力、判定窗口、镜头控制问题
- `wrong_hit` 高：更像识别问题、干扰项设计问题
- `blocked` 高：更像弱点窗口提示不够明确

### 卡 4 进度检查点

用途：判断玩家卡在前段、中段，还是后段收尾。

核心事件：

- `level_checkpoint`

固定检查点：

| 检查点 | 含义 |
|------|------|
| `first_kill` | 第一次击杀成功 |
| `midgame` | 已完成至少一半目标 |
| `endgame` | 只剩最后 2 个目标 |

关键字段：

| 字段 | 含义 |
|------|------|
| `checkpoint_id` | 当前检查点 |
| `killed_targets` | 已击杀目标数 |
| `remaining_targets` | 剩余目标数 |
| `elapsed_time` | 已消耗时间 |
| `remaining_time` | 剩余时间 |
| `scan_used` | 扫描使用次数 |
| `time_extend_used` | 时间延长使用次数 |

判断口径：

- 到不了 `first_kill`：前段识别或基础操作有问题
- 能到 `midgame` 但过不去：中段节奏或移动目标压力有问题
- 能到 `endgame` 但收不掉：后段弱点窗口、时间压力或决策压力有问题

### 卡 5 道具与失败原因

用途：判断玩家有没有被正确“逼出”道具使用，以及第一次主要流失点在哪。

核心事件：

- `item_used`
- `first_failure_recorded`
- `first_exit_recorded`
- `battle_finished`

关键字段：

| 字段 | 含义 |
|------|------|
| `item_id` | 使用的道具类型，当前主要是 `scan`、`time_extend` |
| `remaining_count` | 道具剩余次数 |
| `remaining_time` | 使用时间延长后的剩余时间 |
| `reason` | 第一次失败原因或关卡结束原因 |
| `entry` | 第一次主动退出的入口位置 |

判断口径：

- `scan` 使用率低：先查按钮露出和收益反馈，不要先怪玩家
- `time_extend` 多发生在最后几秒：说明玩家感受到压力，但可能给出时机太晚
- 第一次失败集中在 `时间耗尽`：优先检查时限和后段节奏
- 第一次退出集中在某个面板：优先检查那个节点的信息负担或阻塞感

## 按关卡复盘

### 第 1 关

重点看：

- `tutorial_started -> tutorial_completed`
- `first_hit_elapsed_sec`
- `wrong_hit_count`

优先结论模板：

- “第 1 关问题主要在识别，不在枪法”
- “教程走完了，但首枪成本偏高”
- “误伤高，说明干扰项区分度不够”

### 第 2 关

重点看：

- `midgame` 到达率
- `shot_fired.result = miss`
- `first_failure_recorded.reason`

优先结论模板：

- “玩家能进入中段，但移动目标跟枪成本偏高”
- “失败主要是超时，不是误伤”
- “节奏压力已建立，但时限略紧”

### 第 3 关

重点看：

- `item_used.item_id = scan`
- `item_used.item_id = time_extend`
- `endgame` 到达率
- `battle_finished(success=false)` 的原因分布

优先结论模板：

- “玩家开始使用道具，说明决策层已经被逼出来”
- “很多玩家能到后段，但收尾失败偏多”
- “弱点窗口不是看不见，而是开枪时机偏难抓”

## 本次实际验证结果

本次不是只检查“代码里有没有写事件名”，而是通过 Godot MCP 实际跑了 `integration_smoke_runner`。

### 已确认通过

- 项目可以通过 Godot MCP 启动
- `integration_smoke_runner` 场景可以实际运行
- 新增埋点已真实输出到调试日志：
  - `tutorial_started`
  - `tutorial_completed`
  - `first_hit_recorded`
  - `level_checkpoint`
  - `first_failure_recorded`
  - `battle_finished`
- 第 1 关教程链路与第 2 关失败链路都能打出预期埋点

### 本次为继续测试顺手补掉的问题

为了让烟雾测试能真正跑起来，这轮还顺手补了两个阻塞问题：

1. 缺失 `ui_panel_settings.tscn`，导致主流程预加载失败
2. `PveTargetController.setup()` 里 `Array[String]` 的强类型赋值报错

这两个问题不属于“复盘看板”本身，但会直接拦住自动测试入口，所以已经一起清掉。

### 当前剩余未通过项

本次 `integration_smoke_runner` 最后剩下 1 个失败项：

`将可疑目标移入观察区后，应能看到可疑特征提示`

这个失败说明：

- 测试已经能跑到“观察 -> 聚焦目标 -> 读取搜索提示”这一步
- 当前阻塞点不是埋点系统，而是 `搜索提示文案` 没有按预期出现 `可疑点`

这类问题会影响第 1 关的“识别反馈”质量，但不会影响当前埋点链路本身的有效性。

## 下一步建议

如果继续往下收尾，建议先修这一条：

1. 检查 `_debug_get_search_state()` 返回的 `hint`
2. 检查聚焦可疑目标后，搜索提示是否真的调用了 `get_suspicion_summary()`
3. 保证观察区聚焦到目标时，提示文案里稳定出现 `可疑点：...`

修完这条后，再跑一次 `integration_smoke_runner`，大概率就能接近全绿。
