## 3. 命名规则映射

### 准备层命名规则

当前准备层已统一：

1. 角色：`alien_xxx` / `civilian_xxx`
2. 掩体：`cover_xxx`
3. 反馈：`fx_xxx`
4. 贴花：`decal_xxx`

### 主项目当前映射方式

| 准备层命名 | 主项目当前位置 | 当前状态 | 后续动作 |
|---|---|---|---|
| `alien_*` / `civilian_*` | `pve_target_controller_3d.gd` 的行为与状态变量 | 已可对上 | 后续转配置键 |
| `cover_*` | `pve_cover_obstacle_3d.gd` 的 `style_id` | 已可对上 | 后续转成更一致的资源键 |
| `fx_*` | `visual_feedback_3d.gd` 的反馈语义 | 已可对上 | 后续转成反馈资源表 |
| `decal_*` | 当前主要是掩体内部弹孔逻辑 | 部分对上 | 后续独立 decal 资源键 |

## 4. 接入字段模板映射

### 准备层字段

当前准备层建议字段：

1. `asset_name`
2. `asset_type`
3. `asset_state`
4. `source_stage`
5. `target_node`
6. `trigger_condition`
7. `replace_timing`
8. `fallback_plan`
9. `collision_layer`
10. `mount_points`
11. `notes`

### 主项目未来接入落点建议

| 字段 | 未来建议落点 |
|---|---|
| `asset_name` | 资源表键名 / 配置键 |
| `asset_type` | 资源表分类 |
| `asset_state` | 角色状态、掩体状态、反馈状态的配置字段 |
| `source_stage` | 暂时保留在准备层，不必写入主项目运行时 |
| `target_node` | 对应 `LevelRoot / ActorRoot / DecalRoot / FxRoot / UiRoot` |
| `trigger_condition` | 战斗逻辑、行为逻辑、反馈逻辑 |
| `replace_timing` | 仅文档字段，不必直接进入运行时代码 |
| `fallback_plan` | 资源缺省策略 |
| `collision_layer` | 角色 / 掩体的碰撞层规则 |
| `mount_points` | `HitFocus / WeakPoint / DecalSurface / ImpactFx` 等 |
| `notes` | 文档保留字段 |

## 7. 当前不做的事情

本轮映射明确不做：

1. 不在主项目新增资源表
2. 不把准备层字段写入 `.gd` 文件
3. 不改当前状态变量命名
4. 不建立新的运行时配置文件

---

## 8. 后续真正接入时的最小动作

当后面允许真正接入主项目时，接入字段侧建议按下面顺序推进：

1. 先把 `asset_name` / `target_node` / `trigger_condition` 三项做成最小映射表
2. 再把 `mount_points` 和 `collision_layer` 补进去
3. 最后才考虑把 `replace_timing` / `fallback_plan` 转成更细的资源管理文档

---

## 9. 当前映射结论

接入字段这条线的结论是：

**准备层里的命名、字段、替换顺序和资源状态规则，已经能稳定对到主项目现有的节点层级和脚本职责上。**

也就是说，后面真正开始接入时，不需要重新设计字段体系，只需要把准备层里的字段映射成主项目可读的最小资源表或配置表。
